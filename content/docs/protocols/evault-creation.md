# eVault Creation Protocol

## Purpose

This protocol defines the message exchange required to allocate and initialize a new hosted eVault for a user. It covers provider selection, eName allocation, initial certificate binding, and initialization confirmation.

It does not define:

- recovery setup
- key rotation after initialization
- remote CA issuance
- downstream application onboarding

The provider-side orchestration described here is implemented by the eVault Provisioning Service inside the provider's management boundary.

The meaning of `controlMode`, `subjectKeyState`, and related state markers is defined in [eVault Control Models](/docs/architecture/evault-control-models/).

## Protocol Overview

The protocol consists of two mandatory exchanges and one optional exchange:

1. `Create eVault` request and response
2. `Bind Initial Certificate` request and response
3. Optional `Registry Publish Hosting` update

For delegated variants, certificate binding may be deferred or omitted.

This protocol supports the four creation use cases by carrying an explicit control model declaration instead of inferring behavior from subject type alone. See [eVault Control Models](/docs/architecture/evault-control-models/).

## Protocol Roles

| Role | Description |
|---|---|
| Wallet | Initiates requests, generates key pair, creates certificate, verifies provider responses |
| Provisioning Service | Internal provider-side eVault management component that authenticates initiators for delegated flows, allocates eNames, validates evidence, and orchestrates creation |
| Cloud Provider | Allocates eName, provisions hosted eVault, persists certificate, returns signed receipt |
| Registry | Optional external system for publishing hosting information |

## Control Model Dependency

This protocol must be interpreted together with [eVault Control Models](/docs/architecture/evault-control-models/).

In particular:

- `controlMode` selects the governing control model
- `requestedSubjectType` classifies the subject, but does not by itself determine authorization behavior
- `subjectKeyState` shows whether the subject's key material is bound, deferred, or absent

## Message Envelope

All protocol messages should use a common envelope structure.

```json
{
  "messageType": "string",
  "protocolVersion": "1.0",
  "correlationId": "uuid",
  "timestamp": "2026-04-22T10:15:30Z",
  "sender": {
    "type": "wallet|provider|registry",
    "id": "string"
  },
  "payload": {},
  "signature": {
    "algorithm": "ECDSA_P256_SHA256",
    "keyId": "string",
    "value": "base64url"
  }
}
```

## Common Field Requirements

| Field | Required | Notes |
|---|---|---|
| `messageType` | MUST | Distinguishes protocol message semantics |
| `protocolVersion` | MUST | Enables version negotiation and compatibility rules |
| `correlationId` | MUST | Identifies the end-to-end provisioning flow |
| `timestamp` | MUST | Used for freshness validation and replay protection |
| `signature` | MUST | Every wallet-to-provider and provider-to-wallet message must be signed |
| `payload.eName` | MUST after allocation | Required in every message after provider returns eName |

## Message Sequence

1. Client sends `CreateEvaultRequest` to the provider-side provisioning endpoint.
2. Provider returns `CreateEvaultResponse`.
3. Client sends `BindInitialCertificateRequest` when the selected control model requires subject key binding.
4. Provider returns `BindInitialCertificateResponse`.
5. Provider optionally sends or records `RegistryHostingUpdate`.

### Variant Rules

- Self-provisioned user eVault: all five steps apply.
- Child eVault created by parent: the provider-side provisioning service handles delegated creation; steps 3 and 4 are deferred.
- Organization eVault: the provider-side provisioning service handles delegated creation; steps 3 and 4 may be omitted when no subject signature capability exists.
- Key-owning non-human entity: steps 3 and 4 use subject key material generated by the subject bootstrap environment or key manager.

The exact semantics of these variants are defined by the selected [control model](/docs/architecture/evault-control-models/).

## CreateEvaultRequest

Sent by the initiating client to the provider-side provisioning endpoint to allocate a new hosted eVault.

```json
{
  "messageType": "CreateEvaultRequest",
  "protocolVersion": "1.0",
  "correlationId": "1d5ff7a4-2e92-4bd6-90d9-2f4f24fcf240",
  "timestamp": "2026-04-22T10:15:30Z",
  "sender": {
    "type": "wallet|delegated_client",
    "id": "wallet-instance-123"
  },
  "payload": {
    "requestedSubjectType": "user|child|organization|non_human_key_owner",
    "providerId": "provider-eu-1",
    "controlMode": "self|delegated|governed_without_subject_key|trusted_controller_key_owner",
    "walletNonce": "base64url-random",
    "clientCapabilities": {
      "receiptVerification": true,
      "certificateFormat": ["w3ds-self-signed-v1"]
    },
    "subjectContext": {
      "subjectDisplayType": "natural_person|organization|non_human",
      "subjectKeyExpected": true,
      "delegatedInitiatorEName": "@optional-initiator-ename"
    },
    "evidenceContext": {
      "evidenceRequired": false,
      "evidenceType": "parenthood_document|power_of_attorney|corporate_authorization",
      "evidenceDigest": "base64url-optional"
    }
  },
  "signature": {
    "algorithm": "ECDSA_P256_SHA256",
    "keyId": "wallet-session-key-1",
    "value": "base64url"
  }
}
```

### Validation Rules

- Signature must verify against the initiating client or session verification key.
- `timestamp` must be within the provider's accepted skew window.
- `correlationId` must be unique within the provider's replay-protection window.
- `walletNonce` should be echoed or bound in the response.
- `controlMode` must be semantically consistent with the requested subject type.
- If delegated or governance-based creation is requested, the message must identify whether authority evidence is attached, referenced, or pending.

## CreateEvaultResponse

Sent by provider after eVault allocation.

```json
{
  "messageType": "CreateEvaultResponse",
  "protocolVersion": "1.0",
  "correlationId": "1d5ff7a4-2e92-4bd6-90d9-2f4f24fcf240",
  "timestamp": "2026-04-22T10:15:32Z",
  "sender": {
    "type": "provider",
    "id": "provider-eu-1"
  },
  "payload": {
    "status": "allocated",
    "eName": "@5b8a1c8d-8e7c-4cc9-bde0-21506a69f4ae",
    "evaultId": "c758b522-1d7b-4fc1-b94f-c429f03bc825",
    "evaultState": "pending_key_binding",
    "subjectKeyState": "bound_required|deferred|absent",
    "providerNonce": "base64url-random",
    "walletNonce": "base64url-random",
    "expiresAt": "2026-04-22T10:20:32Z"
  },
  "signature": {
    "algorithm": "ECDSA_P256_SHA256",
    "keyId": "provider-signing-key-2026-01",
    "value": "base64url"
  }
}
```

### Validation Rules

- `eName` must be globally unique within the provider's allocation domain.
- Provider must bind `correlationId`, `walletNonce`, and `providerNonce` to the subsequent certificate-binding step.
- Wallet must verify the provider signature before generating and submitting the certificate.
- The response must explicitly state whether subject key binding is required, deferred, or absent.

## BindInitialCertificateRequest

Sent by wallet after local key generation and certificate creation.

```json
{
  "messageType": "BindInitialCertificateRequest",
  "protocolVersion": "1.0",
  "correlationId": "1d5ff7a4-2e92-4bd6-90d9-2f4f24fcf240",
  "timestamp": "2026-04-22T10:16:10Z",
  "sender": {
    "type": "wallet",
    "id": "wallet-instance-123"
  },
  "payload": {
    "eName": "@5b8a1c8d-8e7c-4cc9-bde0-21506a69f4ae",
    "evaultId": "c758b522-1d7b-4fc1-b94f-c429f03bc825",
    "providerNonce": "base64url-random",
    "walletNonce": "base64url-random-2",
    "certificate": {
      "format": "w3ds-self-signed-v1",
      "subject": {
        "eName": "@5b8a1c8d-8e7c-4cc9-bde0-21506a69f4ae"
      },
      "publicKey": {
        "algorithm": "ECDSA_P256",
        "value": "base64url"
      },
      "issuedAt": "2026-04-22T10:16:05Z",
      "expiresAt": "2027-04-22T10:16:05Z",
      "selfSignature": "base64url"
    }
  },
  "signature": {
    "algorithm": "ECDSA_P256_SHA256",
    "keyId": "wallet-session-key-1",
    "value": "base64url"
  }
}
```

### Validation Rules

- `payload.eName` must exactly match the value returned in `CreateEvaultResponse`.
- The certificate subject must contain the same `eName`.
- The certificate public key and self-signature must be structurally valid.
- `providerNonce` must match the value from the provider allocation response.
- The request must arrive before `expiresAt`.

### Applicability

- Required for self-provisioned user eVaults.
- Deferred for child eVaults until child key material exists.
- Optional or unsupported for organization eVaults without subject signature capability.

## BindInitialCertificateResponse

Sent by provider after certificate persistence and state transition.

```json
{
  "messageType": "BindInitialCertificateResponse",
  "protocolVersion": "1.0",
  "correlationId": "1d5ff7a4-2e92-4bd6-90d9-2f4f24fcf240",
  "timestamp": "2026-04-22T10:16:12Z",
  "sender": {
    "type": "provider",
    "id": "provider-eu-1"
  },
  "payload": {
    "status": "active",
    "eName": "@5b8a1c8d-8e7c-4cc9-bde0-21506a69f4ae",
    "evaultId": "c758b522-1d7b-4fc1-b94f-c429f03bc825",
    "evaultState": "active",
    "recoveryState": "recovery_not_configured",
    "subjectKeyState": "bound|deferred|absent",
    "hosting": {
      "providerId": "provider-eu-1"
    },
    "receipt": {
      "receiptId": "1f7c8c6e-f7ef-4a34-9d9e-0c5f15ec8ab7",
      "issuedAt": "2026-04-22T10:16:12Z",
      "summary": "Initial certificate stored and eVault activated"
    }
  },
  "signature": {
    "algorithm": "ECDSA_P256_SHA256",
    "keyId": "provider-signing-key-2026-01",
    "value": "base64url"
  }
}
```

### Validation Rules

- Response signature must verify against the provider's published verification key.
- Response must include the same `correlationId` and `eName`.
- Wallet should persist success only after signature verification succeeds.

## Deferred or No-Key Completion

When the subject key is not bound during creation, the provider still returns a signed completion payload that makes the control model explicit.

```json
{
  "status": "allocated_or_active",
  "eName": "@5b8a1c8d-8e7c-4cc9-bde0-21506a69f4ae",
  "evaultId": "c758b522-1d7b-4fc1-b94f-c429f03bc825",
  "evaultState": "pending_key_binding|active",
  "recoveryState": "recovery_not_configured",
  "subjectKeyState": "deferred|absent",
  "controlMode": "delegated|governed_without_subject_key"
}
```

## Evidence Handling for Delegated Variants

Delegated creation may require evidence documents that prove the initiator's authority to create the eVault.

Examples:

- parenthood or guardianship evidence for child eVault creation
- power of attorney for organization eVault creation
- a chain of corporate authorization documents

The protocol must support one of the following patterns:

1. Evidence included directly in the provisioning request
2. Evidence uploaded to a controlled evidence service and referenced by digest and identifier
3. Evidence validated by the provider-side provisioning service, with the provider receiving an internal signed attestation or evidence reference

Minimum evidence metadata:

```json
{
  "evidenceId": "uuid",
  "evidenceType": "parenthood_document",
  "digest": "base64url",
  "submittedAt": "2026-04-22T10:12:00Z",
  "validatedBy": "provider-provisioning-service-1",
  "validationState": "accepted|rejected|pending_manual_review"
}
```

## Mapping to the Four Creation Use Cases

| Use Case | Subject Type | Control Mode | Key State at Creation |
|---|---|---|---|
| Create eVault for Self | `user` | `self` | `bound` |
| Create eVault for Non-Subject Entity | `child` | `delegated` | `deferred` |
| Create eVault for Organization | `organization` | `governed_without_subject_key` | `absent` |
| Create eVault for Key-Owning Non-Human Entity | `non_human_key_owner` | `trusted_controller_key_owner` | `bound` |

For normative meaning of these control modes, see [eVault Control Models](/docs/architecture/evault-control-models/).

## Error Response

Any failed step should return a signed error response.

```json
{
  "messageType": "ProtocolError",
  "protocolVersion": "1.0",
  "correlationId": "1d5ff7a4-2e92-4bd6-90d9-2f4f24fcf240",
  "timestamp": "2026-04-22T10:15:31Z",
  "sender": {
    "type": "provider",
    "id": "provider-eu-1"
  },
  "payload": {
    "code": "INVALID_SIGNATURE",
    "message": "Signature verification failed",
    "retryable": false
  },

    "algorithm": "ECDSA_P256_SHA256",
    "keyId": "provider-signing-key-2026-01",
    "value": "base64url"
  }
}
```

## Security Considerations

### Transport Security

- All protocol messages must be sent over HTTPS or an equivalent mutually authenticated secure transport.
- Provider identity should be authenticated through a trustable PKI or equivalent infrastructure trust mechanism.

### Replay Protection

- Every request and response must include `timestamp` and `correlationId`.
- Provider should enforce bounded clock skew.
- Provider should maintain replay detection for `correlationId`, nonces, and signed message digests within a configured retention window.

### MITM Resistance

- Transport protection alone is insufficient.
- Wallet must verify signed provider responses.
- Provider must verify signed wallet requests.
- The certificate-binding request must include the provider-issued nonce from allocation response.

### Certificate Substitution Prevention

- The provider must verify that:
  - the eName in the request matches the allocated eName
  - the eName in the certificate subject matches the allocated eName
  - the request is bound to the same provisioning session
- Provider must reject any attempt to bind a certificate created for a different eName.

### Deferred or Missing Subject Key

- The protocol must make `bound`, `deferred`, and `absent` states explicit.
- Systems must not infer subject signing capability merely from eVault existence.
- Authorization for delegated or organization eVaults must rely on explicit controller or governance metadata.
- Evidence of authority must be integrity-protected, auditable, and linked to the provisioning session.

### eName Handling

- eName is an identifier, not a secret.
- Authorization decisions must not rely on knowledge of eName alone.

### Receipt Integrity

- Provider should return a signed receipt in the final confirmation.
- Wallet should verify and persist this receipt to support auditability and dispute resolution.
